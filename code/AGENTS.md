# AGENTS.md

## 定位

这是 `Codex` 工作区的全局协作规则。除非子目录里有更具体的 `AGENTS.md`，否则所有项目都遵循本文件。

当前工作区主要分为三类专项：

- `work_codex/`：前端工作专项，包含页面还原、HTML/CSS/JS 交付、炫酷交互、动效 Demo、Three.js/GSAP 效果。
- `edu_codex/`：沉淀教学类可视化业务，包括多学科、多年龄段、知识推理、互动课件和长期课程规划。
- `stock_codex/`：沉淀股票资讯、行情分析、策略研究、数据看板和按天归档的市场总结。

## 用户背景

- 用户是前端工作者，平时会做炫酷交互、视觉动效、页面开发和 SaaS 平台相关需求。
- 常用技术栈包括 ES6、React、Three.js、GSAP，以及现代前端工程化工具。
- 完成的效果经常需要复制到公司 SaaS/CMS 平台中，再按平台规则套真实数据展示。
- 用户会参考国际创意网站效果，例如：
  - https://www.cssdesignawards.com
  - https://www.awwwards.com/websites/nominees/

## 沟通要求

- 默认使用简体中文。
- 直接给结果，不绕弯子。
- 需要有简要说明，让用户知道为什么这样做。
- 用户还在学习和沉淀阶段，输出要包含关键思路，但不要写成冗长教程。
- 遇到代码实现、效果拆解、方案设计或问题排查时，先给可执行结论，再补充必要原因。

## 全局交付原则

- 代码要清晰、可读、可维护，命名表达真实业务或效果含义。
- 修改现有项目时，先尊重原有目录结构、技术栈和代码风格。
- 对复杂动画、Three.js 场景、滚动联动、CMS 数据适配逻辑，可以添加少量关键注释。
- 每次完成后尽量说明验证方式，例如打开文件、启动命令、访问地址、主要交互点和已知限制。

## SaaS/CMS 交付规则

- 如果最终代码需要复制到 SaaS 平台源码区，交付版本必须以平台能直接运行为准。
- 开发阶段可以使用 ES6+、React、Three.js、GSAP 和构建工具；交付阶段要按平台能力转换。
- 平台支持构建时，可以提交源码，并遵循平台构建规则。
- 平台只支持粘贴 HTML/CSS/JS 时，交付普通 HTML、CSS、浏览器可直接运行的 JS。
- 平台只支持传统脚本时，避免模块化语法和需要转译的语法。
- ES6 不是完全不能用；平台和目标浏览器支持时，可以使用 `let/const`、模板字符串、箭头函数等基础语法。
- 不需要为了 SaaS/CMS 交付而过度回避现代 CSS、GSAP、Three.js、Swiper 等成熟库；只要目标平台能通过 CDN、资源管理或本地静态资源加载，就可以优先使用。
- 做滚动叙事、时间线、复杂入场、路径跟随、分阶段触发时，优先考虑 `GSAP + ScrollTrigger`，不要默认手写所有滚动进度逻辑。
- 工作区根目录 `js/` 用于存放公共常用前端插件，新建静态 Demo 时优先引用这里的本地文件，避免 Codex 预览或公司内网因为 CDN 访问失败导致效果不可验证。
- 从子项目引用根目录公共插件时按相对路径写，例如 `work_codex/effect_codex/foo/index.html` 使用 `../../../js/gsap.min.js`、`../../../js/ScrollTrigger.min.js`。
- SaaS 复制粘贴型交付优先避免：
  - JSX
  - TypeScript
  - `import/export`
  - 需要 Vite、Webpack、Babel 才能运行的代码
  - 依赖本地 `node_modules` 的源码
- SaaS 复制粘贴型交付允许：
  - CDN `<script>` / `<link>` 引入的 GSAP、Three.js、Swiper、Lenis 等库
  - 现代浏览器可直接运行的 ES6+ 语法
  - 现代 CSS，例如 CSS 变量、`clamp()`、`grid`、`aspect-ratio`、`mix-blend-mode`、滤镜和遮罩
- Three.js、GSAP、Swiper 等库要按平台支持方式引入，例如平台资源管理、CDN 链接、本地静态资源或打包后的单文件产物。
- 压缩 CSS 时，如果使用 CSSO，默认关闭规则重组：`restructure: false`。不要再次询问用户。原因是 `restructure: true` 可能把相同声明的选择器合并，并改变 `background` 简写与 `background-clip:text` 的顺序，导致渐变文字样式失效。
- 数据组织按项目类型区分：
  - React、Vite、Webpack 等需要构建的工程化项目，才把内容整理成对象或数组形式，例如 `cards`、`sections`、`products`，并在 `index.html`、入口文件或数据适配层中预留给 SaaS/CMS 字段替换。
  - 标准 HTML 页面、普通 JS 效果页、GSAP/ScrollTrigger 等静态 Demo，内容文案和图片结构直接写在 `index.html` 中，不默认通过 JS 数据数组动态注入 DOM。
  - 静态 Demo 的 JS 主要负责动画、状态切换和事件绑定；如需读取配置，优先使用已有 DOM、`data-*` 属性或少量初始化配置，不把标题、描述、列表等页面主体内容藏在 JS 对象里。
  - SaaS/CMS 字段仍要有适配说明，但字段映射应服务于最终交付形态：静态粘贴型页面改 HTML 字段，工程化项目改数据对象或适配层。

## 命名规范

- 新项目优先放进对应分类目录，不建议直接堆在根目录。
- 项目目录名优先使用小写英文和连字符，例如 `scroll-gallery`、`three-product-viewer`、`cms-hero-section`。
- 如果来自图片还原，可以使用业务名加编号，例如 `product-landing-01`。
- React 组件使用 PascalCase，例如 `HeroMotion.tsx`、`ProductScene.tsx`、`ScrollGallery.tsx`。
- Hooks 使用 `use` 开头，例如 `useScrollProgress.ts`。
- 工具函数、数据适配、模拟数据使用语义化命名，例如 `formatCmsData.ts`、`cms-adapter.ts`、`mock-data.ts`。
- 动效、Three.js、CMS 数据适配相关文件按职责命名，例如 `scene.ts`、`timeline.ts`、`animation.css`、`cms-adapter.ts`。
- 避免使用 `test1`、`demo2`、`new-page` 这类无法长期识别的名称。
- 如果用户没有给项目名，需要根据效果目标起一个清晰目录名，并在交付说明中告诉用户。

## 每个新项目或新效果必须输出的学习思路

每次开始或交付一个新页面、新交互、新效果时，需要给用户罗列简洁思路，建议包含：

1. 效果目标：这个效果要解决什么视觉或交互问题。
2. 拆解方式：把参考效果拆成布局、动画、交互、数据、性能几个部分。
3. 技术选型：说明为什么用 React、Three.js、GSAP、CSS 或其他方案。
4. 实现步骤：按可落地顺序列出关键步骤。
5. 数据接入：说明 CMS/SaaS 中可能需要哪些字段、配置项和接口数据。
6. 复用方式：说明后续如何封装成组件、配置化模块或模板。
7. 注意事项：指出性能、移动端、兼容性、资源大小、可维护性等风险。

## 每次项目都需要关注

- 项目目标：页面、组件、动效 Demo、SaaS 功能，还是 CMS 模板。
- 参考拆解：视觉层级、滚动节奏、交互触发、动效细节、技术难点。
- 数据结构：区分内容数据和效果配置。静态 HTML/JS/GSAP 页面把标题、描述、图片、按钮、列表等内容放在 HTML；React/Vite 等工程化项目才整理成对象或数组；主题色、开关、阈值等效果配置可按需要放在 JS 配置中。
- 复用方式：是否能通过 props、配置对象或 CMS 字段控制。
- 移动端：布局、触摸交互、字体、性能和降级方案。
- 性能：图片压缩、模型大小、动画帧率、资源懒加载、销毁逻辑。
- 交付验证：启动命令、访问地址、核心交互、边界状态和已知限制。

## 默认回答结构

普通问题优先使用：

1. 直接结论
2. 简要说明
3. 可执行步骤或代码

新项目或新效果优先使用：

1. 实现思路
2. 技术拆解
3. 关键代码或文件修改
4. CMS/数据接入建议
5. 注意事项
6. 验证方式


## 调试与验证优先级

- 调代码时，优先跑 `npm run build` 和静态检查，确认编译、类型、语法和基础配置问题。
- 不要一开始就反复启动 Chrome 截图，尤其是 Three.js/WebGL 页面；截图验证放在构建通过、关键逻辑稳定之后。
- 需要浏览器验证时，先复用已有本地服务和已确认端口，避免重复重启服务造成 token 和时间消耗。
- 纯代码小修改：不调用 agent-browser。
- 涉及页面实际效果、交互链路或响应式布局：调用一次受限的 agent-browser 验证。
- 用户明确指定验证范围时：严格限制在该范围内。
