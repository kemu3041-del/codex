# Text Orbit Motion 项目复盘

## 1. 本次效果结论

这个项目最终实现的是 3D 圆柱文字环：文字不是一整句被弯成椭圆，也不是多个普通文字简单围成椭圆，而是参考图片轮播圆环的结构，把每个图片卡片替换成带弧度的文字面片，再按圆柱面分布。

最终结构：

- React + Three.js + @react-three/fiber + Drei。
- 每个文字块保持面片形态，使用 CanvasTexture 绘制文字。
- 自定义弯曲面片模拟参考示例里的 bentPlane 卡片弧度。
- 按 Y 轴 360 度环绕分布，形成前后纵深。
- 中间红色触手穿过圆环，前后文字和中心物体形成遮挡关系。
- CMS 字段集中在 `src/cms-adapter.js`，支持文字、数量、半径、主题色和运动参数。

## 2. 为什么前期会反复拉扯

主要原因不是技术实现不了，而是提示词里缺少几个决定性约束：

1. “文字环绕运动”容易被理解成普通 2D 椭圆轨道，或者整句文字沿路径变形。
2. “一句文字，拉伸成椭圆”和“多个文字”是两个不同方向，模型会在路径文字、重复文字、卡片圆环之间来回切换。
3. 参考图的关键不是椭圆本身，而是“多个平面元素围绕 Y 轴组成圆柱环，并有前后纵深”。
4. 示例代码里最关键的是 `position=[sin(angle)*radius, 0, cos(angle)*radius]` 和 `rotation=[0, PI + angle, 0]`，以及 `bentPlaneGeometry`。没有这个结构提示时，很容易做成平面布局。
5. “后面文字消失/变黑”和“后面文字灰色可见”是两种完全不同的深度表现，需要明确写死。
6. 镜头也影响判断：需要说明镜头略高、看向中心、圆环视觉接近椭圆，而不是几何上把圆环压成 2D 椭圆。

## 3. 一步到位提示词模板

后续做类似效果，可以直接这样写：

```text
创建一个 React + Three.js + @react-three/fiber + Drei 的 3D 文字圆环效果。

目标不是 2D 椭圆路径文字，也不是把一句文字弯曲成一整条路径；
目标是参考图片轮播 carousel ring 的结构：多个独立的文字面片围绕 Y 轴组成一个圆柱形 3D ring。

实现要求：
1. 重复短语 "Content" 或 "TEXT"，围绕 360 度分布。
2. 每个文字块是独立面片，文字本身保持可读，不要拆成单个字母。
3. 每个面片要像 Drei Image + bentPlaneGeometry 那样有轻微弧度，不是完全平面卡片。
4. 圆环位置公式：x = sin(angle) * radius，z = cos(angle) * radius。
5. 每个面片绕 Y 轴旋转，正面朝圆柱外侧，rotationY = Math.PI + angle 或按实际朝向修正。
6. 圆环半径控制在 2 到 3 左右，不要太大，要能看到后排文字。
7. 镜头略高于圆环，看向中心，让圆环视觉上呈椭圆透视。
8. 中心有一个红色有机触手或产品模型穿过圆环中心。
9. 文字环必须能看出前后纵深：前排文字白色，后排文字灰色可见，不要黑色消失。
10. 使用 depthTest 让中心物体和文字产生真实遮挡，但不要让后排文字完全不可见。
11. 保留 CMS 配置：ringPhrase、ringCopies、ringRadius、theme、motion.speed、motion.tilt。
12. 代码中给关键数学、Three.js 布局、CMS 适配逻辑写简短注释。

请先实现可运行页面，不要做营销落地页；首屏直接展示这个 3D 效果。
```

如果已经有参考代码，要追加：

```text
参考代码中的 Carousel/Card 是核心参考：保留它的环绕位置、Y 轴旋转、bentPlaneGeometry 思路；忽略 Banner 外层丝带；把 Image 替换成文字纹理面片。
```

## 4. 本项目后续复用建议

- 继续保留 `src/cms-adapter.js`，不要把 CMS 字段散落到 Three.js 组件里。
- `ringPhrase` 改文字，`ringCopies/ringRadius` 可手动控制；不填时自动计算。
- `motion.speed` 控制圆环旋转速度，`motion.tilt` 控制圆环倾斜角。
- 如果接 SaaS/CMS，优先把字段映射到 adapter，再传给场景组件。
- 如果要换中心物体，把 `OrganicCore` 替换成产品模型或 GLTF，文字环逻辑不需要改。

## 5. 服务请求失败和重启的原因

这次多次服务请求失败，主要不是业务代码问题，而是验证环境和权限叠加导致：

1. 当前项目本身没有安装 Playwright/Puppeteer 浏览器运行时。虽然 Node REPL 里能找到 Playwright 包，但它自己的 Chromium 没下载，所以首次 Playwright 验证失败。
2. 系统 Chrome 可以被调用，但 headless Chrome 对 WebGL 截图不稳定，容易出现画布背景有了、主体没截出来的情况。
3. Codex 沙箱限制会影响本地端口监听、访问 `127.0.0.1`、启动 Chrome、读取调试端口等操作，所以有时需要提升权限。
4. 开发服务器端口被占用、服务已停止、或者沙箱内 `curl` 无法访问本地端口时，会造成重复启动和重复验证。
5. WebGL 页面验证比普通 DOM 页面更容易触发重试，因为“页面能打开”和“3D 内容真的渲染出来”是两件事。

## 6. 如何避免以后消耗 token

推荐流程：

1. 开始前先确认验证方式：用 Vite 本地预览、Codex 内置浏览器、系统 Chrome，还是用户自己打开地址确认。
2. 如果只是调代码，优先跑 `npm run build` 和静态检查；不要一开始就反复启 Chrome 截图。
3. 如果必须做 WebGL 截图，先确认是否已有可用浏览器工具；没有就不要反复尝试 Playwright。
4. 开发服务器只启动一次，记录端口；后面只检查已有服务，不反复重启。
5. 遇到沙箱限制时，先说明原因，再一次性申请所需权限，不要多条命令连续试错。
6. 对 WebGL 效果，截图失败不一定代表代码失败；要区分构建错误、运行时错误、headless 截图限制。

## 7. 关于 Codex 沙箱设置

截图里的“完全访问”能减少一部分问题，但不能解决所有问题。

它主要影响：

- 是否能写文件。
- 是否能启动本地服务。
- 是否能访问本地端口。
- 是否能调用系统 Chrome 或调试端口。

它不能自动解决：

- 项目没有安装 Playwright 浏览器。
- Chrome headless 对 WebGL 截图不稳定。
- 端口已被旧服务占用。
- 页面代码本身运行时异常。

建议：

- 做本地前端/Three.js 项目时，可以把 Codex 权限设置到允许工作区写入和本地服务访问，能减少 EPERM 和重复审批。
- 不建议为了所有任务长期无脑完全访问；涉及删除、git reset、系统目录写入时仍然要谨慎。
- 对这种 3D 效果，最省 token 的方式是：我完成代码和构建验证，你在浏览器直接打开本地地址做最终视觉确认；只有视觉不对时再继续调。
