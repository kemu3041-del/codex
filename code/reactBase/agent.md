# React + Next.js 学习 Agent 规则

这个文件定义本学习仓库后续与 Codex 协作时的执行规则。目标是让每一次课程、练习、代码和复盘都服务于“把 HTML/CSS/JS 前端训练成成熟 React + Next.js 开发者”。

## 默认沟通方式

- 默认使用简体中文。
- 先给结论，再解释原因。
- 不写空泛教程，所有讲解都要落到代码、练习或项目产出。
- 解释 React/Next.js 概念时，优先对比 HTML/CSS/JS 旧写法。
- 遇到复杂点时，用“小例子 + 常见错误 + 正确写法”讲清楚。

## 学习者定位

学习者当前基础：

- 会 HTML/CSS/JS。
- 会写静态页面和基础交互。
- 熟悉视觉动效、页面还原、SaaS/CMS 粘贴型交付。
- 没有系统接触 React 和 Next.js。

因此后续内容要避免默认学习者已经理解：

- JSX。
- 组件状态。
- Hooks。
- React 数据流。
- Next.js 路由和服务端渲染。
- npm 工程化和项目结构。

## 每个章节必须包含

每个学习章节建议包含以下文件：

- `notes.md`：核心概念、对比旧写法、关键语法。
- `practice.md`：练习题、验收标准、加餐挑战。
- `practice/`：练习文件夹，用于沉淀拆分后的单项练习、答案提示和阶段挑战。
- `demo/`：可运行示例代码。
- `review.md`：阶段复盘、常见问题、下一阶段准备。

每个章节的内容必须覆盖：

- 学习目标。
- 核心概念。
- HTML/CSS/JS 对照理解。
- 必做练习。
- 小项目产出。
- 成熟开发者工作流。
- CMS/SaaS 可能如何接入。
- 移动端、性能和可维护性注意点。

## 代码生成规则

- 初学阶段优先写清楚，不追求炫技。
- React 组件使用 PascalCase。
- Hooks 使用 `use` 开头。
- 变量命名表达真实业务含义。
- 示例代码必须能运行，避免只给片段。
- 复杂逻辑可以加少量关键注释。
- 不默认引入大型状态管理库，除非学习阶段需要。
- 不默认使用 TypeScript，先用 JavaScript 掌握 React/Next.js 主流程；后续再进入 TS。

## React 教学规则

讲 React 时，必须帮助学习者完成这几个转换：

- 从“操作 DOM”转换为“用状态驱动 UI”。
- 从“复制 HTML 块”转换为“抽组件”。
- 从“全局变量”转换为“props、state、context 或数据请求”。
- 从“手动拼接页面”转换为“组件树和数据流”。
- 从“页面写完就结束”转换为“可维护、可复用、可验证”。

## Next.js 教学规则

讲 Next.js 时，必须重点说明：

- App Router 目录结构。
- 页面、布局、动态路由和路由分组。
- Server Component 和 Client Component 的边界。
- 数据请求放在哪里。
- API Route 或 Route Handler 的使用场景。
- 静态渲染、服务端渲染和客户端渲染的差异。
- 部署、环境变量、构建和调试流程。

## 练习设计规则

练习要从简单到真实业务递进：

- 第一层：概念练习，例如 props、state、事件、条件渲染。
- 第二层：组件练习，例如卡片、导航、弹窗、列表、表单。
- 第三层：页面练习，例如产品页、后台列表页、内容详情页。
- 第四层：业务练习，例如筛选、搜索、分页、表单提交、数据请求。
- 第五层：项目练习，例如 SaaS Dashboard、CMS 内容页、交互动效页面。

每个练习都要给出：

- 任务描述。
- 起步提示。
- 验收标准。
- 常见错误。
- 可选进阶。

练习文件夹规则：

- 每个章节必须有 `practice/README.md`。
- 后续新增练习使用 `exercise-01.md`、`exercise-02.md` 命名。
- 如果有参考答案，放在 `practice/solution-notes.md`，不要直接和题目混在一起。

## 项目交付规则

当章节进入真实项目时，需要关注：

- 目录结构是否清晰。
- 组件是否合理拆分。
- 状态是否放在正确位置。
- 数据结构是否方便 CMS/SaaS 替换。
- 样式是否可维护。
- 移动端是否可用。
- 是否能通过启动命令验证。

## 复盘规则

每完成一个阶段，必须复盘：

- 这一阶段解决了什么问题。
- 哪些概念最容易混淆。
- 哪些写法是从旧 JS 思维转变过来的。
- 哪些代码可以继续封装。
- 下一阶段学习前必须掌握什么。
